Sensa Risk Intelligence AML GA Integration Test Plan

Author: Ciaran Finnegan | Date: February 11, 2026

Objective: Provide a structured, repeatable integration test strategy and execution steps for SRI AML GA data flows.

Integrated Test Plan Strategy

This section documents the integrated test strategy for the SRI AML GA release, aligned to the approved architecture.

The focus is on batch-only processing, MinIO-based ingestion, and the core flows between Sensa Data, Detection, Investigation, and Agents.

Scope and Objectives

  • Validate end-to-end batch data ingestion, AML inference, and investigation workflows across Sensa Data, Sensa Detection, Sensa Investigation, and Sensa Agent.
  • Confirm integration points between MinIO feeds, OLAP/OLTP stores, Kafka events, and investigation search.
  • Demonstrate the AML GA release operates in batch mode only and uses MinIO as the sole source feed.

Architecture Summary

  • MinIO feeds are ingested by Sensa Data (Data Connectors) and validated by Data Quality before landing in OLAP (Iceberg) and OLTP (PostgreSQL).
  • Sensa Detection ingests transactions directly from MinIO for AML Inference (batch) and uses OLAP data for training.
  • Decisioning Service enriches detection events with OLTP data and publishes Detection Events to Kafka.
  • Sensa Investigation consumes Subject Updated and Detection Events from Kafka and queries OLTP for search.
  • Sensa Agent capabilities (Operational Assistants and Investigator Copilot) are accessed from Investigation.

Test Approach

  • Exercise each integration flow with batch payloads aligned to the SRI AML GA IFS schema.
  • Validate positive flows and error handling (data quality failures, missing references).
  • Capture evidence from logs, Kafka topics, and UI/API responses for traceability.

Environments and Test Data

  • Use non-production MinIO buckets with controlled batch payloads (transactions, subjects, accounts).
  • Seed OLTP reference data required for enrichment and investigation search.
  • Use Kafka topics dedicated to test execution to isolate events and ensure clean assertions.

Entry and Exit Criteria

  • Entry: Batch ingestion endpoints available, Kafka topics provisioned, and OLAP/OLTP accessible.
  • Exit: All integration test cases executed with expected results and defects triaged.

Integration Flows Under Test

  1. MinIO -> Sensa Data (Data Connectors) -> Data Quality -> OLAP/OLTP
  2. MinIO (direct) -> AML Inference (batch) in Sensa Detection
  3. Data Quality -> AML Inference trigger
  4. OLAP (historic data) -> Sensa Training
  5. AML Inference -> Decisioning Service -> Kafka Detection Events
  6. Sensa Data -> Kafka Subject Updated Events -> Sensa Investigation
  7. Sensa Investigation -> OLTP Transaction Search
  8. Sensa Investigation -> Sensa Agent (Operational Assistants, Investigator Copilot)

Out of Scope

  • Real-time ingestion or streaming processing.
  • Data ingestion from sources other than MinIO.
  • Ontology-driven access for AML inference or investigation search.
  • Entity Resolution, Watchlist Management, and Data Mapping Agent.

Assumptions

  • SRI AML GA IFS schema is the agreed format for test data.
  • Kafka topics and schemas are available for Subject Updated and Detection Events.
  • OLTP contains required reference data for Decisioning enrichment.

Day Zero: Sanity Integration Tests

Short sanity checks to confirm core batch ingestion and integration points are operational.

These tests are designed to be fast and unblock Day 1 and Day 2 execution.

DZ-01: Environment and connectivity smoke check

Purpose: Confirm core infrastructure endpoints are reachable before executing batch integrations.

Traceability: MinIO / Kafka / OLAP / OLTP readiness

Test data notes:

  • Use service account credentials for MinIO, Kafka, and DBs.
  • Prepare a unique test run ID for logging correlation.
  1. Authenticate to MinIO and list ingestion buckets for Sensa Data and Sensa Detection.
    Expected: MinIO access succeeds and required buckets are listed without permission errors.
    Suggested data format: N/A (connectivity check).
  2. Publish and consume a sample message on the Subject Updated and Detection Events Kafka topics.
    Expected: Kafka round-trip succeeds with no schema errors.
    Suggested data format: Minimal JSON event payload with run ID and timestamp.
  3. Run a simple OLAP and OLTP query for seeded reference data.
    Expected: Queries return expected results and no connectivity or permission issues are seen.
    Suggested data format: SQL query output or Iceberg partition metadata.

DZ-02: Minimal batch ingestion through Data Quality

Purpose: Verify a single small batch can be ingested, validated, and stored in OLAP/OLTP.

Traceability: MinIO -> Sensa Data -> Data Quality -> OLAP/OLTP

Test data notes:

  • SRI AML GA IFS schema with 1-3 transactions and subjects.
  • Include deterministic IDs for reconciliation.
  1. Upload a minimal batch file to the MinIO ingestion bucket.
    Expected: Sensa Data registers an ingestion job and picks up the batch without delay.
    Suggested data format: CSV or Parquet file aligned to SRI AML GA IFS.
  2. Review Data Quality results for the batch.
    Expected: Data Quality reports a pass status with no mandatory-field errors.
    Suggested data format: Include mandatory fields only, no optional errors.
  3. Confirm OLAP and OLTP record counts match the batch payload.
    Expected: Stored counts and key fields match the source.
    Suggested data format: Record-level reconciliation using source IDs.

DZ-03: Direct transaction feed to AML Inference (batch)

Purpose: Ensure Sensa Detection can ingest a direct transaction batch from MinIO and execute inference.

Traceability: MinIO -> AML Inference (Batch)

Test data notes:

  • Transactions must match SRI AML GA IFS schema.
  1. Place a transaction batch file in the Sensa Detection MinIO feed location.
    Expected: AML Inference job is triggered for the batch.
    Suggested data format: Transactions file with timestamps, amounts, and counterparty identifiers.
  2. Verify inference outputs are produced and tagged with the batch run ID.
    Expected: Each transaction receives a risk score or label.
    Suggested data format: Detection output records with transaction IDs and risk scores.

DZ-04: Decisioning publishes Detection Events to Kafka

Purpose: Validate Decisioning enrichment and Detection Event publishing for a minimal inference run.

Traceability: Decisioning -> Kafka Detection Events

Test data notes:

  • OLTP reference data must exist for enrichment fields.
  1. Confirm Decisioning retrieves OLTP enrichment data for the inference output.
    Expected: Detection events include enriched subject and account attributes.
    Suggested data format: OLTP reference rows for subject and account IDs.
  2. Consume the Detection Events topic for the run.
    Expected: Kafka messages are published and schema valid.
    Suggested data format: Detection event JSON with risk score and transaction ID.

DZ-05: Investigation consumes detection events (sanity)

Purpose: Confirm Sensa Investigation can ingest Detection Events from Kafka and create a basic investigation record.

Traceability: Kafka Detection Events -> Sensa Investigation

Test data notes:

  • Use a single detection event with required fields.
  1. Publish a valid Detection Event to Kafka.
    Expected: Sensa Investigation consumes the event without errors.
    Suggested data format: Detection event JSON with transaction ID, risk score, and subject reference.
  2. Locate the created investigation record.
    Expected: An investigation case exists and references the detection event ID.
    Suggested data format: Case record showing event ID and initial status.

Day 1: Detection to Investigation and Agent Integration

Integration testing focused on Sensa Detection, Sensa Investigation, and Sensa Agent interactions.

Validates Kafka event consumption, investigation workflows, and AI assistant engagement.

D1-01: Detection event creates investigation case

Purpose: Validate end-to-end flow from inference output to investigation case creation.

Traceability: Decisioning -> Kafka -> Sensa Investigation

Test data notes:

  • Use a batch with multiple transactions and varied risk.
  1. Run AML Inference for a transaction batch and capture Decisioning output.
    Expected: Decisioning emits Detection Events for each high-risk transaction.
    Suggested data format: Transactions file with risk-relevant attributes.
  2. Verify Sensa Investigation consumes Detection Events and creates cases.
    Expected: Investigation cases are created with correct risk scores and transaction references.
    Suggested data format: Detection event JSON with event ID and score.
  3. Confirm case details match Detection Event data.
    Expected: Case fields align with event metadata and subject attributes.
    Suggested data format: Case view data exported for verification.

D1-02: Subject Updated Events refresh investigation context

Purpose: Ensure subject updates from Sensa Data appear in investigation views.

Traceability: Sensa Data -> Kafka -> Sensa Investigation

Test data notes:

  • Update subject attributes used in investigation display.
  1. Ingest an updated subject record and monitor the Subject Updated Events topic.
    Expected: A Subject Updated event is published with the updated attributes.
    Suggested data format: Subject update event JSON with subject ID and change summary.
  2. Verify investigation case context reflects the updated subject details.
    Expected: Subject fields in the case match the updated data.
    Suggested data format: Case context snapshot before and after update.

D1-03: Investigation transaction search via OLTP

Purpose: Confirm investigation search queries OLTP and returns complete transaction details.

Traceability: Sensa Investigation -> OLTP Transaction Search

Test data notes:

  • Use transactions ingested in the same test cycle.
  1. Execute a transaction search in Investigation using known transaction IDs.
    Expected: Search results return matching transactions with full details.
    Suggested data format: Search criteria: transaction ID, date range, subject ID.
  2. Compare the search output with OLTP records.
    Expected: Field-level values match the OLTP source data.
    Suggested data format: OLTP query output for the same transaction IDs.

D1-04: Case lifecycle updates and audit trail

Purpose: Validate case workflow actions are tracked and do not break event traceability.

Traceability: Sensa Investigation workflow

Test data notes:

  • Use a case created from a detection event in D1-01.
  1. Update the case status through key workflow states (e.g., New -> In Review -> Escalated).
    Expected: Status changes are persisted and visible across sessions.
    Suggested data format: Case update payload with status, user, timestamp.
  2. Review the audit trail for status changes and linked detection events.
    Expected: Audit trail shows all changes and references the event ID.
    Suggested data format: Audit log export with case ID and timestamps.

D1-05: Operational Assistant guidance in investigations

Purpose: Ensure Operational Assistants provide workflow guidance aligned to the investigation context.

Traceability: Sensa Investigation -> Sensa Agent

Test data notes:

  • Use a case with multiple alerts and subject details.
  1. Open a case and request Operational Assistant guidance for a standard task.
    Expected: Assistant response references the case data and suggests next actions.
    Suggested data format: Case summary with alert IDs and subject details.
  2. Record the response and verify it includes the correct transaction and subject references.
    Expected: Assistant output aligns with case evidence and does not omit key alerts.
    Suggested data format: Assistant response transcript for evidence.

D1-06: Investigator Copilot analysis on case data

Purpose: Validate Investigator Copilot produces insights grounded in the case context.

Traceability: Sensa Investigation -> Sensa Agent

Test data notes:

  • Use a case with transaction history and risk indicators.
  1. Request a case summary from Investigator Copilot.
    Expected: Summary reflects case facts and risk indicators from the data.
    Suggested data format: Case dataset with transaction timeline.
  2. Compare Copilot response to source case evidence.
    Expected: No conflicting or missing key facts are found.
    Suggested data format: Evidence checklist with expected highlights.

Day 2: Full End-to-End Integration Tests

Comprehensive end-to-end integration coverage across all Sensa Risk Intelligence applications.

Includes full batch processing, enrichment, investigation, and agent assistance with volume and error scenarios.

D2-01: End-to-end batch flow across all applications

Purpose: Validate full SRI AML GA integration from ingestion to investigation and agent assistance.

Traceability: MinIO -> Sensa Data -> Detection -> Kafka -> Investigation -> Sensa Agent

Test data notes:

  • Use a full SRI AML GA IFS batch with subjects, accounts, and transactions.
  1. Upload a full batch to MinIO and confirm Sensa Data ingestion and Data Quality completion.
    Expected: Data Quality completes with a pass rate recorded.
    Suggested data format: CSV or Parquet batch with subject, account, transaction entities.
  2. Verify AML Inference and Decisioning publish Detection Events to Kafka.
    Expected: Detection Events include enrichment fields and correlation IDs.
    Suggested data format: Detection event JSON with enriched attributes.
  3. Confirm Investigation cases are created and Operational Assistants can respond.
    Expected: Cases exist with alerts, and assistant responses match case context.
    Suggested data format: Case export with alert IDs and subject details.

D2-02: Model training uses OLAP history and updates inference

Purpose: Ensure Sensa Training consumes OLAP history and the resulting model is used for inference.

Traceability: OLAP -> Sensa Training -> AML Inference

Test data notes:

  • OLAP contains historic data for at least one period.
  1. Initiate a training run for a historic date range.
    Expected: Training job reads OLAP partitions and completes with a new model version.
    Suggested data format: OLAP partitions with date-based transaction data.
  2. Run inference and confirm the model version used matches the latest training output.
    Expected: Inference metadata references the updated model.
    Suggested data format: Model artifact metadata with version and hash.

D2-03: Data Quality failure prevents inference trigger

Purpose: Validate failure handling when data quality rules fail.

Traceability: Data Quality -> AML Inference trigger

Test data notes:

  • Include invalid records that violate mandatory rules.
  1. Upload a batch with missing mandatory fields.
    Expected: Data Quality marks the batch as failed.
    Suggested data format: Batch file with missing subject identifiers.
  2. Confirm AML Inference is not triggered for the failed batch.
    Expected: No inference job is created and errors are logged.
    Suggested data format: Data Quality failure report with error codes.

D2-04: Cross-system correlation and traceability

Purpose: Ensure a batch run ID is traceable across all systems.

Traceability: Batch ID across Data, Detection, Kafka, Investigation

Test data notes:

  • Use a unique batch ID embedded in file and event metadata.
  1. Execute a batch with a unique run ID and capture Data Quality and inference metadata.
    Expected: Run ID is present in Data Quality, inference, and decisioning metadata.
    Suggested data format: Batch filename and metadata containing run ID.
  2. Trace the run ID through Kafka events and investigation cases.
    Expected: Detection Events and case records include the same run ID.
    Suggested data format: Event payload with correlation ID field.
